# End-to-End QA Run — June 2026

**Method:** Local dev server (`npm run dev`, `NEXT_PUBLIC_BYPASS_PAYMENT=true`) on `http://localhost:3000`, driven with browser automation. Persona drafts were seeded into the persisted Zustand store (`lastminute-itr-draft`) to drive each screen with realistic numbers, then screens were captured and the on-screen figures verified against the tax engine.

**Personas exercised live:** CS-1 (salaried happy path), CS-2 (job-change multi Form 16), CS-3 (AIS-vs-Form 16 mismatch). Remaining personas (CS-4..CS-8) are validated via the typed fixtures + engine (`lib/filing/__tests__/personas.test.ts`) and the simulation matrix; full click-through deferred to keep the run focused.

**Fixtures:** [e2e/fixtures/personas.ts](../../e2e/fixtures/personas.ts) · **Case studies:** [../case-studies/CASE_STUDIES_2026.md](../case-studies/CASE_STUDIES_2026.md) · **Pain points:** [../research/USER_PAINPOINTS_2026.md](../research/USER_PAINPOINTS_2026.md)

## Headline result

The funnel is functionally solid and the **tax math is correct and consistent across every screen** (import → parsing → review → regime → checkout). Companion-mode compliance copy is strong throughout. The biggest opportunities are **clarity/navigation**, not correctness — exactly what Phase 2 targets.

### Cross-page number consistency (verified)

| Persona | Gross | TDS | Old regime | New regime | Shown consistently on |
|---------|-------|-----|------------|------------|------------------------|
| CS-1 Aarav (₹12L, HRA, 80C/80D) | 12,00,000 | 85,000 | Refund ₹21,456 | Refund ₹85,000 (FY25-26 ≤₹12L rebate → 0 tax) | review summary rail, review/summary tab, regime |
| CS-2 Priya (₹13L combined, no HRA) | 13,00,000 | 97,000 | tax ₹1,41,336 | Refund ₹71,000 (tax ~₹26k) → saving ₹1,15,336 | parsing rail, mismatch, checkout "earned value" |

CS-1 new-regime refund equals the full TDS because, under FY2025-26 new-regime rules, taxable income ≤ ₹12,00,000 is effectively zero-tax after the 87A rebate. This is correct, not a bug.

## Prioritized defect / gap backlog (scopes Phase 2)

### P0 — drives Phase 2
1. **Reconcile clarity on `/file/review`.** The #1 pain point (AIS vs Form 16 vs 26AS) is not a first-class dashboard on the main review screen; reconciliation power is hidden on `/file/import/mismatch`. → **p2-review-dashboard**: refund/payable hero + side-by-side AIS/Form 16/26AS cards with matched/needs-attention states + deduction checklist, reusing `lib/filing/reconciliation.ts`.
2. **"Where am I / distance to filing" is implicit.** The Path bar exists (`ProductProcessFlow`) but lacks explicit completed/current/upcoming affordances + section status. → **p2-path**.

### P1 — navigation & consistency
3. **Two nav systems.** Marketing `SiteHeader` vs filing `FilingLayout`, plus a mobile bottom tab bar AND a side "Filing sections" nav AND the Path bar — the funnel has 3 simultaneous navigation surfaces. → **p2-nav**: unify into one calm premium system with shared tokens; reduce redundancy.
4. **Button/card/token drift.** `components/ui/button.tsx` (marketing) vs `components/filing/ui.tsx` (filing). → **p2-polish**.

### P2 — copy / verify
5. Parsing "Latest Form 16" card shows aggregate, not the latest single employer — fix label or value.
6. "Demo parsing" banner on a real/extracted parse — confirm demo detection only fires for `demo_fallback`.
7. Companion page duplicated "Filing progress" / "What this means" headings — verify single render.

### Non-issues confirmed
- Tax math, regime recommendation, multi-employer aggregation, value-before-pay gating, and compliance copy are all correct.
